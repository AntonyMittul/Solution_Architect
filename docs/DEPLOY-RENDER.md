# Deploying to Render

The repo ships a Render Blueprint (`render.yaml`) that provisions everything:

| Component | Render type | Notes |
|---|---|---|
| `aisa-web` | Web Service (Docker) | The only thing users hit. Proxies `/api/*` to the API at runtime. |
| `aisa-api` | Web Service (Docker) | FastAPI. Runs `alembic upgrade head` on boot. |
| `aisa-worker` | Background Worker (Docker) | Runs the intake + blueprint agent graphs. |
| `aisa-postgres` | Postgres | System of record. RLS enforces tenant isolation. |
| `aisa-redis` | Key Value (Redis) | Job queue, SSE pub/sub, per-run token meter. |

## Cost — read this first

A **fully free** deploy is not possible for this architecture:

- **Background Workers have no free plan** on Render. The worker runs every agent,
  so without it intake and blueprint runs stay queued forever.
- **Free Postgres expires after 30 days.**

Budget roughly a starter worker + a small Postgres. Redis has a usable free tier.
If you only want a demo, you can temporarily set `AISA_LLM_PROVIDER=fake` so no
LLM spend occurs — the whole app still works with placeholder content.

## Steps

1. **Push to GitHub** (already done) and make sure `render.yaml` is on your default branch.

2. **Render → New → Blueprint**, connect the repo. Render reads `render.yaml` and
   shows the five resources. Adjust plan names if Render's slugs have changed.

3. **Fill the secrets** it prompts for (`sync: false` values):
   - `AISA_GEMINI_API_KEY` — on **both** `aisa-api` and `aisa-worker`.
     (Or set `AISA_LLM_PROVIDER=fake` on both and leave the key blank.)
   - `AISA_MCP_AUTH_TOKEN` — only if you set `AISA_MCP_CLIENT=http`.
   - `API_URL` on `aisa-web` — you cannot know this until the API deploys, so:

4. **First deploy** → let `aisa-api` finish, copy its URL
   (`https://aisa-api-xxxx.onrender.com`), set it as `API_URL` on `aisa-web`,
   and redeploy `aisa-web`.

5. **Verify**:
   - `GET https://<api>/health/ready` → `{"status":"ok","postgres":"ok","redis":"ok"}`
   - Open the web URL, register, click **Resend / verify** on the banner
     (no SMTP is wired — verification completes in-app).

`AISA_SECRET_KEY` is generated by Render (`generateValue: true`). The app refuses
to boot outside dev without it, and cookies automatically become `Secure`.

## What the code does for you

- **DSN normalisation.** Render hands out `postgres://…?sslmode=require`. The app
  rewrites it to `postgresql+asyncpg://…` and strips `sslmode` (asyncpg rejects it).
  Prefer Render's **Internal Database URL** — it stays on the private network.
- **Migrations tolerate managed Postgres.** Migration `0002` creates a restricted
  `aisa_app` role, but managed users often lack `CREATEROLE`. It now skips the role
  instead of aborting; the app then connects as the database owner and
  `FORCE ROW LEVEL SECURITY` keeps every policy applying to the owner too.
- **RLS self-check at startup.** If the runtime role can bypass RLS (a superuser, or
  `BYPASSRLS`), the API logs `db.rls_bypassed` at ERROR. Tenant isolation would then
  rest on the application layer alone. **Check your logs for `db.rls_enforced` after
  the first deploy.** If you see the error, create a non-superuser role and point
  `AISA_DATABASE_URL` at it (keep `AISA_DATABASE_ADMIN_URL` on the owner for migrations).
- **Runtime proxy, not build-time rewrites.** `API_URL` is read per request by the
  route handlers, so the same image promotes across environments. Server-Sent Events
  stream unbuffered through a dedicated handler.

## Operational notes

- **Scaling.** `aisa-api` and `aisa-web` are stateless — scale horizontally freely.
  Scale `aisa-worker` on queue depth; runs are checkpointed and idempotent, so a
  worker restart resumes rather than duplicates.
- **Zero-downtime.** Migrations are expand/contract, so old pods keep working while
  a new revision rolls out.
- **Backups.** Enable Postgres backups in the Render dashboard. Redis is
  reconstructible (jobs re-derive from the `runs` table; SSE replays from `agent_events`).
- **Cost control.** Plan quotas and the per-run token budget are enforced in-app.
  Defaults: 50 runs/month and 500k tokens/run on the free plan
  (`AISA_FREE_MONTHLY_RUN_QUOTA`, `AISA_FREE_RUN_TOKEN_BUDGET`).
- **Observability.** Set `AISA_OTEL_ENABLED=true` and `AISA_OTEL_ENDPOINT` to ship
  traces and metrics to a collector.

## Not wired yet

Email (SMTP), billing, and GitHub OAuth are deliberately not configured. Email
verification is completed in-app via **Resend / verify**; there is no password reset.
Treat this as a strong internal/beta deployment rather than a public signup product
until SMTP and billing land.
